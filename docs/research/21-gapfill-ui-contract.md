# Gap-fill: UI streaming/data contract

*Condensed findings (agent-researched 2026-07-21, verified against live sources). Key facts and sources only; full reports available on request.*

## KEY FACTS
- @langchain/react@1.0.28 is v2-native: no streamMode/joinStream/reconnectOnMount/onCustomEvent; it speaks the thread-centric LangGraph Agent Streaming Protocol (@langchain/protocol@0.0.18) via POST /threads/{id}/commands and POST /threads/{id}/stream/events (SSE, or WebSocket on the same path), with channels values|updates|messages|tools|lifecycle|input|checkpoints|tasks|custom:<name>.
- useStream returns { values, messages (BaseMessage[]), toolCalls, interrupts/interrupt, isLoading, isThreadLoading, hydrationPromise, error, threadId, subagents, subgraphs, subgraphsByNode, submit, stop, disconnect, respond, respondAll, client, assistantId, getThread } — resume is stream.respond()/respondAll(), NOT submit(null,{command}).
- UseStreamOptions is a discriminated union: AgentServerOptions { assistantId (required), apiUrl?, apiKey?, client?, defaultHeaders?, transport?: 'sse'|'websocket', fetch?, webSocketFactory?, maxReconnectAttempts?, streamIdleReconnect?, reconnectDelayMs?, onReconnect?, threadId?, onThreadId?, onCreated?, onCompleted?, initialValues?, messagesKey? ('messages'), tools?, onTool?, optimistic? (default true) } | CustomAdapterOptions { transport: AgentServerAdapter }.
- Message streaming is content-block-centric: message-start → content-block-start/delta/finish per block → message-finish; deltas are text-delta, reasoning-delta, data-delta (base64), block-delta (shallow merge, used for tool_call_chunk partial-JSON args); content-block-finish upgrades tool_call_chunk to tool_call with parsed args; all wire fields snake_case.
- Tool execution arrives on the 'tools' channel as tool-started {tool_call_id, tool_name, input} → optional tool-output-delta → tool-finished {output} | tool-error {message}; the client assembles AssembledToolCall { name, callId, id (alias), namespace, input, args (alias), output (null until success), status 'running'|'finished'|'error', error }.
- Deepagents todo item is identical in Python 0.6.12 and JS 1.11.1: { content: string, status: 'pending'|'in_progress'|'completed' } at values.todos, written by the write_todos tool.
- Files state diverges by language: Python FileData = { content: string, encoding: 'utf-8'|'base64', created_at?, modified_at? }; JS FileData = FileDataV1 { content: string[] (lines), created_at, modified_at } | FileDataV2 { content: string|Uint8Array, mimeType, created_at, modified_at }; both use Record<path, FileData> with null-delete updates (Python via DeltaChannel reducer).
- HITL casing verdict: Python langchain HumanInTheLoopMiddleware emits snake_case {action_requests:[{name,args,description}], review_configs:[{action_name,allowed_decisions,args_schema?}]}; JS langchain humanInTheLoopMiddleware emits camelCase {actionRequests:[{name,args,description}], reviewConfigs:[{actionName,allowedDecisions,argsSchema?}]}; the SDK's normalizeHitlInterruptPayload (applied automatically in the v1 controller) aliases both casings with camelCase canonical — but actionName/action_name and argsSchema/args_schema on reviewConfigs are NOT aliased, so read both.
- HITL resume payload: { decisions: [ {type:'approve'} | {type:'edit', editedAction:{name,args}} (Python: edited_action; SDK duplicates both on send) | {type:'reject', message?} | {type:'respond', message} ] }, one decision per interrupted call in order; sent via stream.respond(response) → protocol input.respond { namespace, interrupt_id, response, update?, goto?, config?, metadata? } (batch: {responses:[...]}); reject yields ToolMessage status='error', respond yields status='success'.
- SubagentDiscoverySnapshot = { id (task tool_call_id), name (subagent_type), namespace (segments 'tools:<toolCallId>', legacy 'task:'), parentId, depth, status 'running'|'complete'|'error', taskInput (description), output, error, startedAt, completedAt }; discovered eagerly from root 'tools'/'values' events; content is lazy via useMessages/useToolCalls(stream, snapshot) which ref-count namespaced subscriptions through ChannelRegistry.
- The always-on root subscription (ROOT_PUMP_CHANNELS) covers ['values','checkpoints','lifecycle','input','messages','tools'] at namespace []; interrupts hydrate from thread.getState().tasks[].interrupts and live-update from input.requested events {interrupt_id, payload}.
- agent-chat-ui (main, July 2026) still uses the LEGACY @langchain/langgraph-sdk@1.8.10 react hook with options {apiUrl, apiKey, assistantId, threadId, fetchStateHistory:true, onCustomEvent, onThreadId}; its effective streamMode is ['values','updates'] plus auto-added 'messages-tuple' and callback-driven modes — not ['values'] alone.
- v1 branching/time-travel replaces getMessagesMetadata/setBranch with: checkpoints channel envelopes {id,parent_id,step,source}, useMessageMetadata(stream,id).parentCheckpointId, submit(input,{forkFrom:checkpointId}), and state.listCheckpoints/state.fork commands; optimistic messages expose optimisticStatus 'pending'|'sent'|'failed'.
- The current docs HITL page still shows the legacy stream.submit(null,{command:{resume}}) pattern which does not exist on the v1 submit options type — the UI spec must standardize on respond()/respondAll(); headless client tools use a separate interrupt shape {type:'tool', toolCall:{id,name,args}} auto-handled when tools[] is registered.
- Managed Deep Agents (private beta, LangSmith Cloud US-only) compiles a project dir via the mda CLI into a standard hosted LangGraph deployment whose Agent Server exposes the protocol-v2 endpoints, so the same @langchain/react client contract applies unchanged.

## OPEN QUESTIONS
- Whether the hosted LangSmith Agent Server enables the WebSocket upgrade on /threads/{id}/stream/events for all deployment tiers during the Managed Deep Agents beta (SSE is confirmed; WS is documented in the API reference but no tier matrix was found).
- Whether the 'checkpoints' and 'debug' channels are accepted by the deployed Agent Server SSE endpoint despite the API reference enum omitting them (protocol 0.0.18 and the SDK root pump still use 'checkpoints'; could not test a live server).
- Server-side status of multitaskStrategy 'interrupt'/'enqueue'/'reject' on run.start — the SDK types accept all four but note only 'rollback' (plus client-side 'enqueue' queuing) is honoured client-side today, referencing internal roadmap items (plan-roadmap.md §5.3 R2.3, A0.3).
- Exact deepagents version at which JS FileDataV2 (mimeType) replaced V1 (lines array) and whether Python plans to converge on the mimeType shape — the Python 0.6.12 wheel still uses {content, encoding}.
- Whether the docs' human-in-the-loop page submit(null,{command:{resume}}) example still functions against a v1 @langchain/react client via some untyped passthrough (the typed options and submit-coordinator source show no command handling, so it appears to be doc staleness, but this was not verified against a live server).
